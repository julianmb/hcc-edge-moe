# HCC Next-Gen Optimizations (TODO)
*Date: May 15, 2026*

Here are five advanced, research-backed improvements for the HCC framework to push the boundaries of edge LLM inference on unified memory systems:

### 1. Semantic-Aware Co-Scheduling (Sem-MoE)
Currently, `orchestrator.rs` sends the full draft tree to Node 2 for verification. However, MoE routing is highly predictable based on the semantic context. 
*   **The Idea:** Implement a lightweight "Routing Predictor" on the NPU (Node 1). Before generating the draft, it predicts which experts will be activated. 
*   **The Code Change:** If we find that Node 2's iGPU currently lacks the necessary experts in its VRAM (or fast LPDDR5x region), we can trigger an asynchronous prefetch of those expert weights from SSD/Host to the iGPU *while* the NPU is still drafting the tree. 

### 2. "Double-Dovetail" Dynamic Compute Shifting
We currently have an `hcc` pipeline (NPU->GPU) and a `dovetail` pipeline (GPU->CPU).
*   **The Idea:** Strix Halo has a powerful Zen 5 CPU alongside the iGPU. When the iGPU is verifying a massive draft tree, it can become compute-bound. 
*   **The Code Change:** Implement a dynamic load balancer in `orchestrator.rs`. If the draft tree exceeds a certain width/depth, dynamically offload the verification of specific branches to the Zen 5 CPU (using AVX-512/VNNI) while the iGPU verifies the main branches. This creates a true heterogeneous triplet: NPU (draft) -> iGPU + CPU (parallel verify).

### 3. Native RoCEv2 (rocSHMEM) Implementation
We stubbed out the `KernelBypassBackend::RoceV2` logic in `af_xdp.rs`, but it's currently a mock. 
*   **The Idea:** Moving from AF_XDP (17µs) to RoCEv2 (10µs) using AMD's `rocSHMEM` would completely eliminate the OS from the data path.
*   **The Code Change:** Write a Rust FFI wrapper around `libibverbs` and `rocSHMEM`. Transition the `DmaBufDescriptor` into a registered Memory Region (MR) and use `IBV_WR_RDMA_WRITE` to let the NPU write draft tokens directly into Node 2's physical memory over the USB4 tunnel without Node 2's CPU ever waking up.

### 4. Continuous Draft Calibration (CDC)
The `calibrator.rs` updates the NPU draft model via LoRA based on rejection sampling. Currently, this happens offline or periodically.
*   **The Idea:** Implement an online Exponential Moving Average (EMA) update for the draft model.
*   **The Code Change:** As Node 2 rejects tokens, it sends the "correct" target logits back to Node 1 via a low-priority background thread. Node 1 uses a tiny, fused HIP kernel to apply a rank-1 LoRA update to the draft model's output head *in real-time* between generation steps. The draft model learns the target model's "mood" mid-sentence, pushing acceptance rates (α) above 0.8.

### 5. Multi-Node (3+) Ring Topology Support
HCC is currently hardcoded for a dual-node (Node 0 and Node 1) setup over a point-to-point USB4 link.
*   **The Idea:** USB4/Thunderbolt supports daisy-chaining. Users could link 3 or 4 ClawRigs to get 384GB or 512GB of unified memory.
*   **The Code Change:** Refactor `usb4.rs` and `orchestrator.rs` to support a ring topology. Implement a pipeline-parallel MoE verification step, where Node 1 drafts, Node 2 verifies layers 0-20, and Node 3 verifies layers 21-40.

### 6. Agentic Orchestration Layer (The "1:1 Ratio" Pivot)
Inspired by Dan McNamara's (AMD) May 2026 thesis on Agentic AI, we must rethink HCC as not just an inference engine, but an **Agentic Orchestrator**. As workloads shift from simple chatbots to autonomous agents, the CPU (orchestration/tool-use) becomes the bottleneck, demanding a 1:1 CPU:GPU ratio. 
*   **The Idea:** Leverage the 16-core Zen 5 CPU on Node 1 specifically for agentic control loops (planning, tool execution, memory retrieval) while the NPU/iGPU handles the heavy speculation.
*   **The Code Change:** 
    *   **Async Tool-Call Sandbox:** Implement a dedicated worker thread pool in `orchestrator.rs` for Python/Bash tool execution.
    *   **Speculation-Aware Prefetching:** While the Zen 5 CPU is executing a tool (e.g., a database query), the NPU drafts potential "continuation scenarios" based on expected results, ensuring the iGPU never sits idle waiting for the CPU orchestration layer.
    *   **Policy & Security Guardrails:** Offload output validation and safety checks to a dedicated "Monitor" task on the Zen 5 cores, running in parallel with the next token's draft phase.
